bin/rails console
      * shortcut: rails c
      * exit by typing exit
      
bin/rails routes
      * Shows all application routes, as configured in config/routes.rb
      
rails new project-name-here [--database=postgresql][--api]
        * the options specify the default database and api only
      
bundle install
        * shortcut: bundle
      
bin/rails db:create
      * verify with psql
      *  - psql -U username -d database-name
      * database name is configured here: config/database.yml
      *  - automatically is set to the name of your project
      
bin/rails db:migrate
      * verify by ensuring db/schema.rb was updated
      * never change the schema directly!
      * use migrations instead to incrementally change the db
      
bin/rails db:seed
      * runs the db/seeds.rb file
      * good for setting up initial application data, like an admin user
      
bin/rails db:drop
      * destroys the database your project is using (removes all data too)
      
bin/rails server
      * shortcut: rails s
      * starts on port that is different from your frontend
      
bin/rails generate [model || migration || controller
        || serializer] [FileName]
      * ex: Generate just a model (and migration):
      * bin/rails g model User name:string phone:string age:integer

      * ex: Generate a migration to add a foreign key author_id column to books table:
      * bin/rails generate migration addAuthorToBooks author:references

      * ex: Generate a migration to remove a (references / foreign_key) column:
      *  bin/rails generate migration removeAuthorFromBooks author:references

      * ex: Generate a migration to remove a (string) column:
      *  bin/rails generate migration removeAuthorFromBooks author:string
      *   - Always remember to:
      *     - check the migration file for accuracy,
      *     - apply your migrations after you've created them!
      
bin/rails db:rollback
      * Attempts to revert the last migration

      * Want to go back (migrate `down`) to a specific version?
      *   - Use db:migrate:down VERSION=timestamp
      *   - ex: bin/rails db:migrate:down VERSION=20191205162704

      * Want to migrate `up` a specific version? In the above command, change down to up!

      
bin/rails db:migrate:status
        * use the `Migration ID` to rollback to a specific version
        * `down` refers to migrations that have _not yet_ been run!
        * `up` refers to migrations that have been run.

        * Run bin/rails db:version to see the current version of your Database
        * This is the last migration applied, and is also at the top of your db/schema.rb

        * In your migration files, `def change` really is just combining up and down methods

        * More from https://guides.rubyonrails.org/active_record_migrations.html:
        * If you run the rails db:migrate:status command, which displays the status (up or down) of each migration,
        * you should see ********** NO FILE ********** displayed next to any deleted migration file which was once executed on a specific environment but can no longer be found in the db/migrate/ directory.
      
bin/rails generate scaffold [ResourceName] [property_name1:property_type1]
        [property_name2:property_type2]
      * ex: Create an Author resource with two columns: first_name and last_name
      *   bin/rails generate scaffold author first_name:string last_name:string
      *   - gives you all files needed for CRUD, including entry in config/routes.rb
      * shorthand: bin/rails g scaffold Post title body:text
      *   - string is the default data type, so the above title will be a string
      
bin/rails rspec spec[/individual/spec/file.rb]
      * run all specs by just running rspec spec
      * run individual specs by specifying a path to the file
